# Passive Safety Software Change Log

当用户要求记录被动安全控制器的软件变更、上传一张截图并希望把信息持续沉淀到飞书表格时，使用本 skill。

## 适用场景

- 用户发送软件/基线后台截图，要求记录控制器软件变更信息
- 需要首次创建飞书电子表格，后续持续追加记录
- 需要对被动安全控制器（如 APSB 等）的软件版本信息做结构化归档
- 需要在写入前检测重复软件信息，并提醒使用者
- 需要把原始标题进一步拆解为结构化字段，而不是只保留整句标题
- 需要把"车型项目"和"项目阶段"拆分为两个独立列记录
- 当截图标题包含多个车型项目时，需要拆分为多行分别记录
- 截图识别时只关注标题内容，以及表格中与 ECU名称、软件ID 所在行相关的信息
- 优先从 ECU 对应行读取业务字段，不再依赖页面顶部辅助信息

## 识别范围约束

在使用当前 skill 识别截图时，**只需要关注以下两部分内容**：

1. **标题内容**
   - 用于提取：车型项目、项目阶段、基线类型、原始标题
2. **表格中目标 ECU 对应行的信息**
   - 必须至少关注：ECU名称、软件ID
   - 并从同一行继续提取与该 ECU 对应的其他字段，如软件版本、供应商ID、硬件零件号、硬件版本、BT版本等

明确忽略以下区域，除非用户特别要求：
- 页面统计卡片（总数、待确认、待集成等）
- 创建人、协作者、发布时间、发布范围等页头辅助信息
- 与目标 ECU 行无关的其他表格行

## 核心读取规则

### 规则 1：标题只负责拆解项目信息
标题内容只用于提取以下字段：
- 车型项目
- 项目阶段
- 基线类型
- 原始标题

### 规则 2：业务字段优先从 ECU 行读取
以下字段**必须优先从目标 ECU 对应行读取**，不要优先参考页面顶部辅助信息：
- 控制器名称
- 软件ID
- 软件版本
- 供应商ID
- 硬件零件号
- 硬件版本
- BT版本

### 规则 3：页面顶部辅助信息默认不用于写表
像以下内容，默认不作为主记录字段来源：
- 创建人
- 协作者
- 计划发布时间
- 发布范围
- 发布车系
- 统计卡片数字

只有在以下情况才可写入备注：
- 用户明确要求保留
- ECU 行内信息缺失，而页面顶部信息对人工追溯有帮助

### 规则 4：同一 ECU 行优先级最高
若标题、页头、备注区与 ECU 行内容不一致：
- 与 ECU 具体软件信息相关的字段，以 **ECU 行** 为准
- 与项目归属相关的字段，以 **标题** 为准

## 分类规则

### 基线类型
标题中出现的基线类型，**只记录为以下三种之一**：

- `电检`
- `MRD`
- `OTA`

示例：
- `电检基线` → 记为 `电检`
- `MRD基线` → 记为 `MRD`
- `OTA基线` / `OTA用途` → 记为 `OTA`

若标题未明确包含以上关键词，则写 `待确认`。

### 项目阶段
标题中的项目阶段，识别并记录为类似以下类型：

- `P2-2`
- `SOP`
- `EP2`
- `OTA9.0.1`
- `PP4`
- `PPV3&PPV4`
- `P2-2&P3`

示例：
- `W05 PP4 电检基线` → 项目阶段：`PP4`
- `W10国内P2-2&P3电检基线` → 项目阶段：`P2-2&P3`
- `OTA9.0.1` → 项目阶段：`OTA9.0.1`

若无法明确匹配项目阶段，则写 `待确认`。

## 标题拆解规则

当页面标题类似：

- `W10国内P2-2&P3电检基线及海外PPV3&PPV4电检基线 ECU版本准备`
- `W05 PP4 电检基线 ECU版本准备`

需要优先拆成结构化字段，而不是仅写成一整句"原始标题"。

### 拆解目标

- `车型项目`
- `项目阶段`
- `基线类型`
- `原始标题`（保留全文，便于追溯）

### 拆解原则

1. 去掉页面尾部无业务意义的固定后缀，例如：
   - `ECU版本准备`
2. 识别基线类型，按照上文分类规则归一化：
   - 包含 `电检基线` → `基线类型=电检`
   - 包含 `MRD基线` → `基线类型=MRD`
   - 包含 `OTA` 且与基线/用途相关 → `基线类型=OTA`
3. 将"车型项目"和"项目阶段"拆分为两个字段单独记录，不能合并在同一列
4. 从标题前半段提取：
   - `车型项目`：车型代码 + 市场/区域，例如 `W10国内`、`W10海外`、`W05`
   - `项目阶段`：阶段信息，例如 `P2-2&P3`、`PPV3&PPV4`、`PP4`、`SOP`、`EP2`、`OTA9.0.1`

### 多车型拆分规则

当标题包含多个车型项目/多个市场/多个阶段时，**必须拆分为多行记录**，每一行对应一个车型项目。

示例：

标题：
`W10国内P2-2&P3电检基线及海外PPV3&PPV4电检基线 ECU版本准备`

应拆成两行：

- 第1行：
  - 车型项目：W10国内
  - 项目阶段：P2-2&P3
  - 基线类型：电检
- 第2行：
  - 车型项目：W10海外
  - 项目阶段：PPV3&PPV4
  - 基线类型：电检

同一张截图中，以下字段在两行中保持一致：
- 控制器名称
- 软件ID
- 软件版本
- 供应商ID
- 硬件零件号
- 硬件版本
- BT版本
- 原始标题
- 记录时间

### 单车型示例

标题：
`W05 PP4 电检基线 ECU版本准备`

拆为：
- 车型项目：W05
- 项目阶段：PP4
- 基线类型：电检

此时只记录一行。

### 无法稳定拆解时

- `车型项目` / `项目阶段` 写 `待确认`
- 同时保留 `原始标题`

## 后续使用流程

### 1. 提取截图信息

优先使用视觉能力识别截图，并结合用户正文说明补足字段。

提取原则：

- 页面主标题优先用于拆解 `车型项目 / 项目阶段 / 基线类型 / 原始标题`
- 表格中**先定位目标 ECU名称 与 软件ID 所在行**
- 所有业务字段应尽量从该 ECU 对应的同一行提取
- 若截图中同一字段有多个值（例如硬件零件号），保留全部值，并使用 `&` 连接
- 若用户文字已给出更明确解释，优先采用用户明确表述
- 无法可靠识别的字段，明确标记为"待确认"，不要编造

### 2. 字段归一化

- `软件版本`、`硬件版本`、`BT版本` 按 16 进制字符串处理，保留原始大写格式
- 示例：`0009` 的下一版是 `000A`，`0A` 的下一版是 `0B`
- 不要把这些版本号当十进制数值解释
- `控制器名称`统一使用截图中的 ECU 名称，如 `APSB`
- `车型项目`示例：`W10国内`、`W05`
- `项目阶段`示例：`P2-2&P3`、`PP4`、`SOP`、`EP2`、`OTA9.0.1`
- `基线类型`统一记录为：`电检` / `MRD` / `OTA`

### 3. 多车型拆分

若标题包含多个车型项目（如"国内"和"海外"），按上述多车型拆分规则，**拆成多行分别记录**。

### 4. 重复检测

在写入前，先查询现有表格记录，按以下组合判断是否重复：

- 控制器名称
- 软件ID
- 软件版本
- 硬件零件号
- 车型项目

若以上关键字段完全一致，则视为重复。

处理规则：

- 若判定重复：
  - 不直接新增重复记录
  - 明确提醒使用者"该软件信息已存在"
  - 返回命中的已有记录摘要（如控制器名称、软件ID、软件版本、车型项目、记录时间）
- 若仅部分字段一致但存在版本或硬件差异：
  - 视为新记录，可正常追加
  - 同时提示使用者存在相似记录，建议复核

### 5. 写入飞书电子表格

始终写入同一个电子表格。

建议使用追加写入，不覆盖历史记录。

每次新增一行或多行（当标题包含多车型时），字段顺序与表头保持一致。

### 6. 排序要求

写入后按以下优先级排序：

1. 第一优先级：控制器名称（升序）
2. 第二优先级：软件ID（升序）
3. 第三优先级：软件版本（高版本在前；若无法可靠自动排序，则先保持追加顺序并提示人工排序）

## 推荐表头

```text
控制器名称 | 软件ID | 软件版本 | 车型项目 | 项目阶段 | 基线类型 | 供应商ID | 硬件零件号 | 硬件版本 | BT版本 | 原始标题 | 记录时间 | 原始备注
```

## 风险与注意事项

- OCR/视觉识别可能误读字母和数字（如 `0/O`、`8/B`、`6/G`），对软件版本、硬件版本、BT版本要谨慎复核
- 当截图分辨率不足或字段遮挡时，不要强行猜测
- 若标题存在多个市场/多个阶段并列描述，需要按多车型拆分规则拆成多行；无法确认时标记待确认
- 识别时不要分散关注无关页面元素，优先锁定标题与目标 ECU 行
- 若飞书表格已按旧表头创建，后续应补充新列或迁移旧数据，避免信息无法结构化利用
